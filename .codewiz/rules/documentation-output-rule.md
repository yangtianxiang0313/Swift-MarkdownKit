# 文档输出规范 Rule

## 核心原则
**最小化文档输出** - 仅在必要时生成文档，最多两篇。

## 规则

### 何时不写文档
- ✅ 简单的代码修改或bug修复
- ✅ 回答问题或提供解释
- ✅ 代码审查或建议
- ✅ 小规模的功能实现
- ✅ 日常开发任务

### 何时写一篇文档
- 📄 创建新的模块或功能模块
- 📄 重要的架构决策或设计方案
- 📄 复杂的实现步骤需要记录
- 📄 API 或公共接口的说明
- 📄 项目级别的重要更新

### 何时写两篇文档（最多）
- 📋 大型功能实现（需要设计文档 + 实现指南）
- 📋 架构重构（需要旧架构说明 + 新架构说明）
- 📋 复杂的集成方案（需要概述 + 详细步骤）

## 执行方式

### 在回答前判断
1. 这个任务是否真的需要文档？
2. 如果需要，是一篇还是两篇？
3. 文档是否能被后续使用？

### 文档命名规范
- 单篇文档：`FEATURE_NAME.md` 或 `IMPLEMENTATION_GUIDE.md`
- 双篇文档：`DESIGN.md` + `IMPLEMENTATION.md`

### 文档内容要求
- 简洁明了，避免冗长
- 包含必要的代码示例
- 清晰的步骤或说明
- 易于查找和参考

## 示例

### ❌ 不需要文档的情况
```
用户: 帮我修复这个 bug
→ 直接修复代码，在回答中说明修复内容
```

### ✅ 需要一篇文档的情况
```
用户: 实现一个新的 Markdown 渲染器
→ 创建 MARKDOWN_RENDERER_GUIDE.md
→ 包含设计思路、API 说明、使用示例
```

### ✅ 需要两篇文档的情况
```
用户: 重构整个动画系统
→ 创建 ANIMATION_DESIGN.md（新架构设计）
→ 创建 ANIMATION_MIGRATION.md（迁移指南）
```

## 检查清单

在生成文档前，问自己：
- [ ] 这个文档是否会被实际使用？
- [ ] 是否有更简洁的方式表达（如代码注释）？
- [ ] 这个文档是否会长期维护？
- [ ] 是否真的需要两篇文档，还是一篇就够？

## 优先级
1. **代码优先** - 代码本身就是最好的文档
2. **注释其次** - 在代码中添加清晰的注释
3. **文档最后** - 只在必要时才写独立文档
